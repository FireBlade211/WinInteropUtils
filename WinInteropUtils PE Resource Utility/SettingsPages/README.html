Settings system

The PE Resource Utility's settings system is a configuration storage system based on a combination of JSON files and UserControls.

Structure

Directories

All configuration-related files are in the SettingsPages or Config folder in the root of the project. The Config folder contains .settings files for each page. The SettingsPages folder contains a Data subfolder containing JSON files for each in-app settings or property panel, which list all the pages contained within. The SettingsPages folder also contains subdirectories for each settings or property panel, that contain the actual pages themselves. The root of the SettingsPages folder also contains the definition of the IConfigPage interface.

Pages

Each page is represented by a UserControl that implements the IConfigPage interface within /SettingsPages/<DIALOG>. The page should also have an associated .settings file within /Config for storing its settings.

The IConfigPage interface

The IConfigPage interface is an interface that all pages should implement. This interface provides the following methods for your page to handle:

  • void ApplyChanges()
    Applies the changes made in the configuration page.
  • void CancelPreview()
    Fires whenever the user presses the Cancel or OK button, or navigates to a different page. This method gives the page an opportunity to undo any previews. For example, if the page shows a live preview of a setting, it can undo it in this method.
  • bool GetSettingsModified()
    Fires whenever the settings dialog needs to query whether the page was modified.

The interface also provides the OnModify event, which you should Invoke whenever the modification flag that is retrieved with GetSettingsModified changes. If you have a lot of settings, it may be useful to make a helper like this:

private void SetModified()
{
    _modified = true;
    OnModify?.Invoke(this, EventArgs.Empty);
}

Creating a page

To create a page:

  1. Identify the JSON file of the settings dialog you want to add the page to within /SettingsPages/Data.
  2. Add the required info for the page to the JSON file. The schema may differ based on which dialog you're adding the page to, based on each page's requirements. Usually, each page should have a UniqueId property. Note it down, as it will be needed later.
  3. Create a new UserControl within /SettingsPages/<DIALOG> and name it <UNIQUE_ID>Page.
  4. Create a new .settings file for your page within /Config and add your settings.
  5. Add the controls you need on the settings page in the designer and wire up their event handlers. For most pages, these event handlers should modify Config.<SETTINGS_FILE_NAME>, but should not call Save.
  1. Implement IConfigPage. If you wired up your event handlers to modify the .settings file, you should call Config.<SETTINGS_FILE_NAME>.Default.Save inside ApplyChanges.
  2. Build and run the application.
  3. Go to the dialog where you added the page. If you did everything correctly, the page should show up and you should be able to click on it to access its contents.
